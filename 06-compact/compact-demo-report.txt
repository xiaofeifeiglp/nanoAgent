================================================================
nanoAgent 演进报告：从 essence 到 compact
================================================================

[文件清单与行数]
  01-essence/agent-essence.py : 112 行
  02-memory/agent-memory.py   : 148 行
  06-compact/agent-compact.py : 243 行

================================================================
一、01-essence（112 行）—— 最简可运行原型
================================================================
* 单层 while True 循环，每轮只调一次 LLM 并执行一个工具调用。
* messages 列表全量发送，无任何压缩或裁剪。
* 只暴露 execute_bash 一个工具，模型是当前 shell 直通的 LLM。
* 缺陷：随着对话增长，messages 越来越长，token 开销线性上升，且一旦
  超出模型上下文窗口就会直接报错。

================================================================
二、02-memory（148 行）—— 引入持久化记忆
================================================================
* 相比 essence 净增 ~36 行，主要改动：
  - 新增 tools: write_memory() / read_memory()
  - 循环开头从 MEMORY.md 读取历史摘要，作为 system prompt 的补充。
  - 每轮结束时调用 write_memory() 把当前会话总结后落盘。
* 本质：跨会话保留信息，但仍是对当前会话内 messages 不做任何处理，
  长会话的 token 膨胀问题没有解决。

================================================================
三、06-compact（243 行）—— 上下文自动压缩
================================================================
* 相比 memory 净增 ~95 行，核心增量是 COMPACT 机制。
* 关键常量：
    COMPACT_THRESHOLD = 8   # messages 长度超过 8 时触发压缩
    KEEP_RECENT      = 4   # 压缩时保留最近 4 条消息
* 触发流程（should_compact -> compact_messages）：
  1. 若 len(messages) <= COMPACT_THRESHOLD，直接返回原列表。
  2. 否则用 find_recent_start() 决定从何处切分前/后两段。
     - 该函数从后往前扫描，先找到 KEEP_RECENT 个 user/assistant
       边界，再回溯到上一个 tool 消息之前，保证不会把"assistant 发
       起 tool_call / tool 返回结果"这一成对结构从中间切开。
  3. 前半段交给本地 LLM（model="qwen2.5-7b-instruct"）做摘要，
     摘要结果作为一条新的 user 消息（标记 _compact_marker）插入，
     后面拼接上保留的最近消息。
* 同时保留并强化了 02-memory 的 MEMORY 流程：先压缩当前上下文，再
  写入 MEMORY.md，兼顾单次会话的 token 控制和跨会话的信息延续。

================================================================
四、演进要点对比
================================================================
  维度            essence        memory              compact
  ----------------------------------------------------------------
  跨会话记忆        无             MEMORY.md            MEMORY.md
  单次会话压缩      无             无                   阈值+摘要
  工具数量          1              3                    5 (+compact)
  模型切换          单一           单一                 主对话云端 / 摘要本地
  防 tool 截断     不需要         不需要               find_recent_start

================================================================
五、核心 takeaway
================================================================
* compact 不是删消息，而是用一条"摘要"消息替换早期历史；
* find_recent_start 是这套机制能稳定工作的关键——它让压缩窗口始终
  落在 user/assistant 边界，从而避免把 function_call / tool 响应拆散；
* 从 112 -> 148 -> 243 的代码量增长里，真正解决 token 膨胀的只有
  compact 这一步；memory 只是把信息搬到了磁盘，并没有减少入参长度。
